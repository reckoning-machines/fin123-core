# Core Concepts

## commit

Persist current model state and execute a deterministic build.

A commit saves the workbook snapshot, evaluates the scalar and table graphs, and persists the results as an immutable run. Each commit produces a `run_id`. Identical inputs always produce identical outputs.

`commit` is the primary action in Terminal Mode. It replaces the separate commit-then-build workflow with a single deterministic operation.

## scenario

A named label over a committed run.

A scenario captures the input parameters and output scalars at the time of save, along with the `run_id` that produced them. Scenarios are not substitutes for commits. They are labels over reality.

Use `scenario save <name>` after a commit. Use `compare <a> <b>` to see what changed between two scenarios — both inputs and outputs — in a structured tabular diff.

## sweep

A one-dimensional parameter experiment.

A sweep runs the same model multiple times while varying one parameter. Each row in the result is a real execution with its own `run_id` and scalar outputs, not a recalculated cell.

```
sweep revenue_growth 0.04 0.06 0.08 0.10
sweep wacc range(0.06, 0.12, 0.01) --outputs value_per_share enterprise_value
```

Results are persisted in `.fin123/sweeps/` and exportable as CSV. Use `--outputs` to select specific output columns for display.

This replaces Excel's Data Table feature with deterministic, auditable execution. Every sweep point can be independently verified.

## grid

A two-dimensional parameter experiment.

A grid runs the same model across two parameters simultaneously, producing a matrix of one output scalar. Each cell is a real execution.

```
grid revenue_growth 0.04 0.06 0.08 vs wacc 0.08 0.09 0.10 --output value_per_share
```

Exactly one output is displayed in the matrix. Maximum 100 cells per grid. Both parameters must be declared editable inputs.

Results are persisted in `.fin123/grids/` and exportable as CSV in tidy long-form.

## draft

A code artifact generated by AI or written manually.

Drafts are stored in `.fin123/drafts/` as immutable objects with metadata (prompt, model, provider, validation status). A draft must be explicitly validated and applied before it can affect workbook builds.

Revisions create new drafts. The original is never mutated. `draft lineage <id>` shows the revision chain.

Lifecycle: `draft` → `validate` → `apply` → `commit` → `build` → `verify`.

## plugin

A Python file in `plugins/` that registers functions for use in the scalar or table evaluation graphs.

Plugins are loaded by the plugin manager during every build. Each plugin is source-validated (forbidden imports, eval/exec, network patterns), imported, and registered. Plugin version and SHA-256 hash are recorded in `run_meta.json` for audit.

Applied drafts become plugins. The apply step writes the artifact to `plugins/`. The build step loads it.
